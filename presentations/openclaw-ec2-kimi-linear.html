Architecture Decisions

Why EC2 + Kimi + Linear?

AWS EC2
Full control over the environment. Persistent compute for cron jobs. Cost-effective for 24/7 operation vs. serverless for long-running agents.
Kimi K2.5
Excellent coding performance. 200K context window. Cost-effective API pricing. Fast inference for interactive workflows.
Why not just use ChatGPT/Claude web?

We needed: scheduled execution, file system access, API integrations, and workflows that persist beyond a chat session. OpenClaw gives us "AI with hands and a calendar."

Infrastructure

EC2 Instance Configuration

Instance Specs

  • Type: t3.medium (2 vCPU, 4 GB RAM)
  • OS: Ubuntu 22.04 LTS
  • Storage: 20 GB gp3 SSD
  • Region: ap-southeast-1 (Singapore)

Software Stack

  • Node.js 22.x (LTS)
  • OpenClaw (latest stable)
  • PM2 (process manager for auto-restart)

Model Configuration

Setting Up Kimi K2.5

Why Kimi K2.5?

  • Coding-optimized: Strong performance on code generation and review
  • 200K context: Can handle large codebases and long conversations
  • Cost-effective: Competitive pricing for API usage
  • Fast: Low latency for interactive workflows

Configuration in OpenClaw

Add to your OpenClaw config:

{ "models": { "default": "kimi-coding/k2p5", "providers": { "kimi-coding": { "apiKey": "${KIMI_API_KEY}", "baseUrl": "https://api.moonshot.cn/v1" } } } }
Pro tip: Store API keys in environment variables, never in config files. Use AWS Systems Manager Parameter Store or Secrets Manager for production.
Project Management

Linear Integration Setup

What We Built

A two-way sync between OpenClaw and Linear that allows:

  • Creating issues from Telegram conversations
  • Getting issue updates via cron jobs
  • Auto-generating standup reports from Linear activity
  • Linking discussions to Linear issues for context

API Configuration

# Linear API Token (stored in env) export LINEAR_API_KEY="lin_api_xxxxx" # In TOOLS.md (workspace config) ### Linear - API Key: $LINEAR_API_KEY - Team: Engineering - Default Project: Platform
Key workflow: Mention an issue ID (e.g., "TEAM-123") in Telegram ‚Üí OpenClaw fetches the issue details ‚Üí Can update status, add comments, or create sub-issues.
Implementation

Linear Integration Code

Here's how we query Linear issues and create summaries:

# ~/.config/openclaw/linear.sh - Linear API helper LINEAR_API_KEY="${LINEAR_API_KEY}" LINEAR_TEAM="ENG" # Fetch recent issues fetch_linear_issues() { curl -s -X POST \ -H "Authorization: $LINEAR_API_KEY" \ -H "Content-Type: application/json" \ https://api.linear.app/graphql \ -d '{"query": "query { issues(filter: { team: { name: { eq: \"' $LINEAR_TEAM'\" } } }, first: 20) { nodes { id title state { name } assignee { name } updatedAt } } }"}' } # Create daily summary generate_standup() { local issues=$(fetch_linear_issues) # Parse with jq, format for Kimi echo "$issues" | jq -r '.data.issues.nodes[] | "- \(.title) [\(.state.name)]"' }
GraphQL queries we use:
  • issues() ‚Äî List issues with filters (team, state, assignee)
  • issue() ‚Äî Get single issue details by ID
  • issueCreate() ‚Äî Create new issue from Telegram
  • issueUpdate() ‚Äî Update status, assignee, or labels

Security Model

Security by Design

üîë Credentials
Env vars only ‚Äî never in code. API keys for Kimi, Linear, Notion stored in environment, never committed.
üñ•Ô∏è Infrastructure
SSH key-only, IP-restricted. No inbound ports needed ‚Äî OpenClaw connects outbound to APIs only.
üì° Data Flow
HTTPS everywhere. Transient processing ‚Äî no persistent chat logs. No PII in our use case.
üîí API Scope
Read-only tokens where possible. Telegram bot whitelists chat IDs. Integration-level Notion access.
Key insight: Outbound-only architecture eliminates a whole class of attacks. Our server calls APIs ‚Äî nothing calls it.
Threat model: Internal tooling with no customer PII. Appropriate for personal/team automation, not Fort Knox ‚Äî but layered and rotatable.
Replication Guide

How to Set This Up Yourself

Step 1: Infrastructure
Launch EC2 t3.medium, install Node.js 22, configure security group
Step 2: OpenClaw
Install OpenClaw, configure Kimi API key, set up Telegram bot
Step 3: Linear
Create Linear API key, test GraphQL queries, build integration scripts
Step 4: Workflows
Write cron jobs, test manually, then schedule. Monitor with PM2 logs
Cost estimate (monthly):
  • EC2 t3.medium: ~$30 (on-demand) or ~$18 (reserved)
  • Kimi API: ~$5-20 (depends on usage)
  • Linear: Free tier sufficient for small teams
  • Telegram: Free

Total: ~$25-50/month for a fully automated AI workflow engine
